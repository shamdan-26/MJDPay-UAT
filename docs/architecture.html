Folder structure

Every feature folder is now PascalCase and follows the same internal shape: one <Feature>Helper.ts, plus whichever of api/, functional/, ui/, archive/ the flow actually needs.

api/ functional/ ui/ archive/ — present where relevant

03

Two fixture lifecycles

Not every suite wants the same session shape, so there are two fixture files rather than one that fights half the codebase.

scope: test

fixtures.ts

A fresh page per test. One lazy fixture per page object — 25 in total, only the ones a test destructures get built.

17 spec files retrofitted
// beforeEach, before → after
loginPage = new LoginPage(page);
// —
test.beforeEach(async ({ page, loginPage: lp }) => {
  loginPage = lp;
  await loginPage.goto(LOGIN_URL);
});
scope: worker

Homepage/HomepageFixtures.ts

One authenticated session per worker, reused across every Homepage file that worker runs — instead of logging in again per file.

05

Standing conventions

  • Serial mode everywhere. Every describe block runs { mode: 'serial' } — specs share session state or need ordering (logout last, etc.).
  • functional/ vs ui/. functional/ owns business logic and outcomes; ui/ owns element/text presence only. Same flow, deliberately split.
  • Toast guard. waitForToastClear absorbs transient background-API error banners after every landing-page navigation, before any assertion.
  • Archive is discovered, not deleted. Specs under */archive/ are retired but still run unless test.skip() is set at the describe level.
  • Failure artifacts on by default. playwright.config.ts now retains trace, screenshot, and video on failure — not just trace.
